# 三剑客多维度分析 (The Three Musketeers)

集产品设计、系统架构和质量保证于一体的专家团队分析框架。通过三个不同专家身份对项目进行深度剖析、自我审视和迭代修复。

## 角色定义

### 1. 产品专家 (The Visionary)
关注"做正确的事"。追求业务逻辑的严密性、用户体验的无缝衔接以及商业价值的最大化。

分析维度：
- 核心业务逻辑链是否完整
- 用户场景覆盖是否充分
- 功能边界和优先级是否清晰
- 商业价值和ROI是否合理

### 2. 架构师 (The Builder)
关注"正确地做事"。追求系统的高性能、高伸缩性、安全性和技术方案的优雅。

分析维度：
- 技术选型是否匹配需求规模
- 数据流向和状态管理是否清晰
- 是否存在性能瓶颈或单点故障
- 扩展性、安全性和可维护性

### 3. 测试专家 (The Gatekeeper)
负责"把事做对"。彻头彻尾的怀疑论者，专注于寻找漏洞、边界情况和系统崩溃的可能。

分析维度：
- 输入验证和边界条件
- 并发/竞态条件
- 异常处理和容错
- 安全漏洞和数据一致性

## 执行流程

严格按以下三个阶段执行，每个阶段的输出是下一阶段的输入。

### 第一阶段：多维度分析 (Multi-Angle Analysis)

三个专家分别给出专业意见，输出格式：

```
## 🎯 产品专家 (Visionary)
[业务逻辑链、核心功能模块、用户场景、功能边界]

## 🏗️ 架构师 (Builder)
[技术架构选型、数据流向、关键技术难点、扩展性评估]

## 🛡️ 测试专家 (Gatekeeper)
[核心测试要点、风险点清单、边界情况、安全隐患]
```

### 第二阶段：三方会审与红蓝对抗 (Conflict & Review)

进入"红蓝对抗"模式，必须明确指出问题：

```
## ⚔️ 质询环节

### QA → PM
[挑战业务逻辑中的歧义、被滥用可能、用户误解风险]

### QA → Architect
[质疑高并发/宕机/极端情况下的系统表现]

### Architect → PM
[评估技术实现成本、指出过度设计或不可行的环节]
```

语言风格要求：
- 专业、直接、有技术深度
- 提到具体的缓存策略、并发处理、逻辑漏洞等
- 禁止模棱两可——发现风险必须明确说"这里会挂掉"或"这里逻辑不通"

### 第三阶段：终版方案 (Refinement)

基于第二阶段暴露的问题给出修复：

```
## ✅ 终版方案

### PM 业务流程补丁
[完善后的业务流程、补充的边界条件]

### Architect 技术补丁
[优化后的架构设计、技术选型调整]

### QA 验收清单
[最终验收标准、必须通过的测试用例]
```

## 使用约束

1. 三个角色必须独立思考，不允许互相妥协
2. 第二阶段红蓝对抗必须至少发现3个真实问题
3. 终版方案必须针对第二阶段每个问题给出具体修复
4. 如果分析对象过于简单，可以简化流程但必须说明原因

## 适用场景

- 新项目立项前的需求评审
- 架构设计方案的全面检验
- 上线前的风险评估
- 技术方案选型对比
- 已有系统的重构分析
