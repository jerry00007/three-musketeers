# 三剑客评审检查清单

## 产品专家 (Visionary) 检查清单

### 业务逻辑
- [ ] 核心业务流程是否有缺失环节
- [ ] 是否存在循环依赖或死锁的业务流程
- [ ] 权限模型是否覆盖所有操作场景
- [ ] 状态机转换是否完整（每个状态的入口和出口）
- [ ] 是否存在不可逆操作但缺少确认步骤

### 用户体验
- [ ] 操作反馈是否及时且明确
- [ ] 错误信息是否可理解（非技术用户）
- [ ] 是否存在用户可能困惑的操作路径
- [ ] 并发操作时的UI状态是否一致
- [ ] 加载/等待状态是否有适当提示

### 功能边界
- [ ] MVP功能范围是否明确
- [ ] 是否存在模糊的需求描述（"尽可能"、"合适时"等）
- [ ] 国际化/多语言需求是否考虑
- [ ] 数据生命周期管理（创建、更新、归档、删除）
- [ ] 数据权限隔离（租户/用户/角色）

---

## 架构师 (Builder) 检查清单

### 系统设计
- [ ] 是否存在单点故障
- [ ] 数据库选型是否匹配数据特征（关系/文档/时序）
- [ ] 缓存策略是否合理（TTL、失效机制、一致性）
- [ ] 是否需要消息队列，选型是否匹配场景
- [ ] API设计是否RESTful/GraphQL，是否符合团队规范

### 性能
- [ ] 是否存在N+1查询
- [ ] 热点数据是否有缓存层
- [ ] 大数据量场景是否有分页/流式处理
- [ ] 是否存在同步阻塞操作影响响应时间
- [ ] 数据库索引是否覆盖高频查询路径

### 可扩展性
- [ ] 水平扩展是否存在障碍（有状态/无状态）
- [ ] 是否存在硬编码的配置或依赖
- [ ] 新功能接入是否需要改动核心模块
- [ ] 第三方服务故障时的降级策略
- [ ] 数据迁移方案是否设计

### 安全
- [ ] 认证授权机制是否完善
- [ ] 敏感数据是否加密存储和传输
- [ ] 是否存在SQL注入/XSS/CSRF风险
- [ ] API速率限制是否合理
- [ ] 日志是否泄露敏感信息

---

## 测试专家 (Gatekeeper) 检查清单

### 输入验证
- [ ] 所有外部输入是否经过校验（类型、长度、格式、范围）
- [ ] 是否存在整数溢出风险
- [ ] 文件上传是否有类型/大小限制
- [ ] JSON/XML解析是否防止注入
- [ ] URL参数是否防止路径遍历

### 并发与竞态
- [ ] 是否存在竞态条件（check-then-act）
- [ ] 分布式场景下是否需要分布式锁
- [ ] 数据库事务隔离级别是否足够
- [ ] 幂等性设计是否覆盖重试场景
- [ ] 并发写入是否有冲突检测

### 异常处理
- [ ] 所有异常是否被捕获并有明确处理
- [ ] 异常时数据是否回滚到一致状态
- [ ] 第三方服务超时/失败时的处理
- [ ] 是否存在空指针/空值访问风险
- [ ] 优雅降级策略是否完善

### 边界条件
- [ ] 空数据/零值/负值处理
- [ ] 极大值/极小值场景
- [ ] 时区/时间边界（跨日/跨月/跨年）
- [ ] 编码问题（Unicode/BOM/特殊字符）
- [ ] 网络中断/弱网场景

## 红蓝对抗常见攻击角度

### QA攻击PM的常用话术
- "如果用户在X步骤中断，系统状态会怎样？"
- "恶意用户能否通过Y方式绕过限制？"
- "这个功能的ROI是否真的值得实现？"
- "A用户和B用户同时操作同一条数据，结果是什么？"

### QA攻击Architect的常用话术
- "数据库主从延迟时，读写是否会出现不一致？"
- "这个接口QPS到10000时，哪个组件先崩？"
- "缓存雪崩/击穿时，是否有保护机制？"
- "部署时如何做到零停机？"

### Architect反击PM的常用话术
- "这个需求的技术复杂度是Z，建议推迟到下一期"
- "这个功能可以简化为X，效果一样但成本减半"
- "当前架构不适合这类需求，需要先做基建"
- "这个指标在实际场景中无法准确度量"
